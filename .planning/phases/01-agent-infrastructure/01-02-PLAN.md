<objective>
Build the core agent runner service and tool system. The agent runner is the ~100-line loop that calls LiteLLM, detects tool calls, executes them against the board REST API with user auth context, and iterates until the LLM produces a final text response. The tool registry defines tool schemas and handlers.

Purpose: This is the AI engine. Without it, agents cannot process requests, call tools, or produce responses. Plan 03 wraps this in an SSE endpoint; Plan 04 consumes it in the frontend.
Output: Working agent loop with tool execution, tested with mocked LiteLLM calls.
</objective>

  <action>
    Create `backend/app/services/agent_runner.py` with two main functions:

    1. `async def run_agent(config: AgentConfig, message: str, user_context: dict) -> str`:
       Non-streaming version for simple use cases.
       - Build messages list: [system_prompt, user_message]
       - If config.context is provided in user_context (from RunAgentRequest.context), append it as context in the user message or as a separate system message.
       - Loop up to config.max_iterations:
         a. Call `litellm.acompletion(model=config.model, messages=messages, tools=get_tools_for_agent(config.allowed_tool_names), tool_choice="auto" if tools else None, temperature=config.temperature)`
         b. If response has no tool_calls -> return message.content (final answer)
         c. If response has tool_calls -> append assistant message, execute each tool via execute_tool(), append tool results as tool messages, continue loop
       - If loop exhausts -> return "Agent reached maximum iterations without completing."

    2. `async def run_agent_streaming(config: AgentConfig, message: str, user_context: dict) -> AsyncGenerator[dict, None]`:
       Hybrid streaming version (per research Pattern S1): non-streaming for tool iterations, stream only the final text response.
       - Yield `{"type": "start", "agent_name": config.name}`
       - Build messages list same as above
       - Track total usage across iterations (prompt_tokens, completion_tokens)
       - Loop up to config.max_iterations:
         a. Call acompletion with `stream=False` to detect tool calls cleanly (avoids tool call accumulation pitfall)
         b. Accumulate usage from response.usage
         c. If tool_calls:
            - Append assistant message to messages
            - For each tool_call:
              - Yield `{"type": "tool_start", "tool_name": tc.function.name, "tool_call_id": tc.id}`
              - Execute tool: `result = await execute_tool(tc.function.name, tc.function.arguments, user_context)`
              - Yield `{"type": "tool_result", "tool_name": tc.function.name, "tool_call_id": tc.id, "result": result, "success": True/False}`
              - Append tool result message
            - Continue to next iteration
         d. If no tool_calls (final response):
            - Make a SECOND call with `stream=True` to stream the final response text (re-send same messages -- the LLM will produce the same final response but streamed)
            - Actually, OPTIMIZATION: if the non-streaming response already has content, yield it as a single text_delta instead of re-calling. Only use streaming for the final call if we want progressive display. Since this is the first implementation, keep it simple: yield the content from the non-streaming call as a single text_delta event. Progressive streaming of the final response can be optimized later.
            - Yield `{"type": "text_delta", "content": msg.content}`
            - Break
       - If loop exhausts, yield `{"type": "error", "message": "Agent reached maximum iterations"}`
       - Yield `{"type": "usage", "prompt_tokens": total_prompt, "completion_tokens": total_completion, "model": config.model}`
       - Yield `{"type": "done"}`

    IMPORTANT implementation notes:
    - Always use `await litellm.acompletion()` (async), never `litellm.completion()` (sync) -- blocking the event loop is a critical pitfall.
    - Import from litellm directly: `from litellm import acompletion`
    - Parse tool_call.function.arguments as JSON string (it comes as a string from LiteLLM)
    - The user_context dict should contain at minimum: {"email": str, "role": str, "user_id": int}
